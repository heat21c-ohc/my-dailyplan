# Daily Plan 팀 작업 지시서

이 문서는 다른 팀이나 에이전트에게 작업을 맡길 때 가장 먼저 확인해야 하는 지시서다. 작업자는 코드 수정 전에 이 문서를 읽고, 아래 문서들을 함께 확인해야 한다.

## 먼저 확인할 문서

1. `PROJECT_PLAN.md`
   - 프로젝트 목표, 성공 기준, 작업 순서 확인

2. `AGENT_RR.md`
   - 본인 역할과 담당 파일 확인
   - 담당 범위 밖 파일 수정 시 `TASK_LOG.md`에 사유 기록

3. `TASK_LOG.md`
   - 작업 시작 전 `In Progress`에 작업 항목 추가
   - 작업 완료 후 `Done`으로 옮기고 변경 파일, 검증 결과, 리스크 기록

## 공통 작업 규칙

- 기존 기능을 임의로 삭제하지 않는다.
- 요청받은 범위 외 리팩터링은 하지 않는다.
- 변경한 파일과 이유를 `TASK_LOG.md`에 남긴다.
- 기능 변경 시 직접 검증하고 결과를 기록한다.
- UI 변경 시 PC/모바일 화면에서 글자 잘림, 입력칸 겹침, 버튼 클릭 가능 여부를 확인한다.
- JavaScript 변경 시 최소한 `node --check app.js`를 실행한다.
- 불확실한 요구사항은 임의 구현하지 말고 가정과 질문을 `TASK_LOG.md`에 남긴다.

## 팀별 작업 지시

### UX/UI 팀

- 담당 파일: `index.html`, `styles.css`
- 목표: 화면 구분감, 가독성, 모바일 사용성 개선
- 검증: PC/모바일에서 텍스트 잘림, 입력칸 겹침, 버튼 접근성 확인

### Frontend 팀

- 담당 파일: `app.js`, 필요 시 `index.html`
- 목표: 추가, 삭제, 드래그, 자동 날짜 입력, 로컬 저장 안정화
- 검증: 행 추가, 삭제, 순서 변경, 새로고침 후 데이터 유지 확인

### QA 팀

- 담당 파일: `TASK_LOG.md`
- 목표: 성공 기준별 테스트 수행
- 검증: 발견 이슈를 `Issue / Suggestion`에 재현 절차와 함께 기록

### 문서화 팀

- 담당 파일: `PROJECT_PLAN.md`, `AGENT_RR.md`, `TASK_LOG.md`, `TEAM_HANDOFF_GUIDE.md`
- 목표: 변경된 정책, 작업 범위, 인수인계 내용 최신화
- 검증: 새 작업자가 문서만 보고 현재 상태와 다음 일을 이해할 수 있는지 확인

## 작업 시작 전 기록 양식

```text
- ID:
  담당:
  상태: In Progress
  범위:
  예상 변경 파일:
  시작 시간:
  확인한 문서:
  리스크/질문:
```

## 작업 완료 보고 양식

```text
- 담당:
- 작업 ID:
- 변경 파일:
- 변경 내용:
- 검증 결과:
- 남은 이슈:
```

## 핵심 운영 원칙

`TASK_LOG.md`를 작업 게시판처럼 사용한다. 작업 시작 전 기록, 작업 중 변경 사유 기록, 작업 후 검증 결과 기록을 반드시 남긴다. 이 3가지를 지키면 팀 간 공유와 인수인계 품질을 유지할 수 있다.
